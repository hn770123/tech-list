# Google Apps Script（GAS）を用いたアプリ開発に関する技術メモ

## 1. ユーザーのメモ要約

*   Google Apps Script（GAS）は、Google Sheetsをデータベースとして利用でき、管理が極めて容易なサービスである。
*   パフォーマンス（実行速度など）は低いものの、一度作成すれば非エンジニアの素人でも運用可能な点が大きな強み。
*   AIによるコード生成が発展している現在、もっと脚光を浴びても良いサービスではないかと感じている。
*   ただし、無料版でWebアプリケーションとしてデプロイした場合、画面に警告バナーが表示される（未ログイン時の挙動も異なる）ため、Googleとしては無料での本番運用を推奨していないように見受けられる。
*   （余談）Microsoft Teamsにも類似のサービス（Power Apps等）が存在するが、仕様のクセが強く、AIの学習データも少ないため比較対象にはならないと感じている。

## 2. AIの補足・明確化

ユーザーの仮説や所感に関する客観的事実の検証と補足を行います。

*   **無料版Webアプリの警告表示に関する検証:**
    *   **ログイン時:** 無料のGoogleアカウント（`@gmail.com`）でGASをWebアプリとして公開し、ブラウザでアクセスすると、画面上部に「このアプリケーションは Google ではなく、別のユーザーによって作成されたものです。」（This application was created by another user, not by Google.）という警告バナーが強制的に表示されます。これはフィッシング詐欺などを防ぐためのセキュリティ仕様であり、非表示にすることはできません。Google Workspaceの有償アカウントで組織内限定公開にした場合には、このバナーは表示されません。ユーザーの「Googleは無料で本番運用させたくないようだ」という仮説は、セキュリティ担保と有償プランへの誘導の観点から合理的と言えます。
    *   **未ログイン時:** Webアプリのアクセス権限を「全員（Anyone）」に設定していても、アクセスするユーザーがGoogleアカウントにログインしていない場合、アプリがユーザーのID情報（メールアドレス等）を要求する設定になっていると、強制的にGoogleのログイン画面にリダイレクトされます。権限を「全員（匿名ユーザーを含む）」にし、かつスクリプトが個人情報にアクセスしない設定であれば未ログインでも閲覧可能ですが、前述の警告バナーは表示されます。
*   **Teamsの類似サービス（Power Apps等）との比較検証:**
    *   Microsoft Power Appsは、ローコード開発プラットフォームとして強力ですが、独自の関数言語（Power Fx）を使用します。Power FxはExcelの数式に似ているものの、特有の概念や制約（コンテキスト変数、コレクションの扱い、委任問題など）があり、「クセが強い」というユーザーの指摘は事実です。
    *   一方、GASのベースは標準的なJavaScript（ECMAScript）です。JavaScriptは世界で最も使用されているプログラミング言語の一つであり、GitHub等の公開リポジトリに膨大なコードが存在するため、LLM（大規模言語モデル）の学習データ量が圧倒的に豊富です。そのため、AIによるコード生成の精度はGASの方が著しく高くなりやすいという事実は、客観的に支持されます。

## 3. 技術的評価

*   **アーキテクチャの評価 (Google Sheets as a DB):**
    *   内部向け業務アプリケーションにおいて、「保守性の高さ」「入力エラー防止」を重視し、リッチなGUIよりも確実な動作を優先するという観点から、Google SheetsをDBとして利用するアーキテクチャは非常に理にかなっています。インフラ管理（サーバーやデータベースの構築・保守）が一切不要であり、常駐サーバー実装と比較して保守コストは限りなくゼロに近いです。データの視認性も高く、バックアップや復元も容易です。
*   **パフォーマンスと制限:**
    *   GASには厳格なクォータ制限（1日のトリガー実行時間、メール送信件数など）や、スクリプトの最大実行時間（通常6分）が存在します。また、Sheets APIへの頻繁なアクセスや同時書き込みは、ロックの競合やパフォーマンス低下（タイムアウトエラー）を引き起こしやすいため、高負荷な処理や大規模なトランザクションには不向きです。
*   **総評:**
    *   低負荷で小規模〜中規模な社内ツール（フォームからのデータ収集、定期的なデータ集計、簡単な承認ワークフローなど）においては、最高クラスのコストパフォーマンスと保守性を発揮します。

## 4. AIコーディングエージェントとの親和性

*   GASはJavaScriptをベースとしているため、AIコーディングエージェントとの親和性は極めて高いです。
*   標準的なJavaScriptの文法に加え、`SpreadsheetApp` や `DriveApp` といったGoogle提供のAPIリファレンスもインターネット上に豊富に存在するため、AIは高精度なGASコードを生成できます。
*   ローカル開発環境（`clasp` や TypeScriptの導入など）を構築すれば、AIエージェントに直接ファイルの生成やデプロイ（`clasp push`）を実行させることも可能であり、開発・運用の自動化のポテンシャルは非常に高いと言えます。

## 5. 補足情報（細かいパターン等）

*   **データアクセス（DBとしての利用）のベストプラクティス:**
    *   セルへの読み書きを1つずつ行うと非常に遅いため、`getValues()` や `setValues()` を用いて、二次元配列としてデータをバルク（一括）で読み書きするのがGASの鉄則です。
    *   同時書き込み時の競合を防ぐため、`LockService` を用いてスクリプトの排他制御（ロックの取得・解放）を実装することが推奨されます。
*   **Webアプリとしての公開パターン:**
    *   `doGet(e)` や `doPost(e)` 関数を実装し、HTMLServiceを利用してWeb画面を返す（SSR: サーバーサイドレンダリング的なアプローチ）か、ContentServiceを利用してJSONを返す（Web API化してフロントエンドはSPAで構築するアプローチ）の2つのパターンが主流です。UIの自由度を高める場合は、GASをJSON APIとしてのみ利用し、フロントエンドは別の静的ホスティング（GitHub PagesやCloudflare Pagesなど）に配置する構成も有効です。
