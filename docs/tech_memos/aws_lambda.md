# AWS Lambda + DynamoDBに関する技術メモ

## 1. ユーザーのメモの要約
AWS LambdaとDynamoDBの組み合わせは低コストでバックエンドを作成するのに適している。ただし特有のクセがあり、業務アプリケーションをLambdaベースで構築すべきか、あるいはより標準的な構成（常駐サーバーなど）を採用すべきかという、AWSサービスの最適な組み合わせを選択するセンスが要求される。結果としてAWS全体に関する包括的な知識が必要となり、学習コストが高くなるという課題がある。一方で、過去に問題となった高額請求に対しては対策方法が確立されており、日本語ドキュメントの充実度や運用ノウハウの集まりやすさという大きなメリットも存在する。

## 2. AIの補足/明確化
ユーザーの指摘する「高額請求の対策」「学習コストと包括的知識」「ドキュメントやノウハウの充実」について、客観的な事実に基づき補足します。

*   **高額請求の対策**: 過去、無限ループや予期せぬトラフィックの急増による高額請求（いわゆる「クラウド破産」）が問題になることがありましたが、現在では対策が確立されています。具体的には、AWS Budgetsを用いた予算超過時のアラート設定、AWS Cost Anomaly Detectionによる機械学習を用いた異常なコストの検知、およびLambdaの同時実行数（Concurrency）の制限設定などを組み合わせることで、意図しない高額請求を未然に防ぐことが可能です。
*   **学習コストと包括的な知識の必要性**: Lambda単体ではなく、API Gateway（HTTPリクエストのエンドポイント）、VPC（プライベートネットワーク内のリソースへのアクセス）、IAM（詳細な権限管理）、CloudWatch（ログと監視）など、周辺のAWSエコシステムと組み合わせた設計が不可欠です。インフラをコード化するIaC（AWS CDK, SAM, Terraformなど）の知識も求められるため、単純なバックエンド開発に比べて初期の学習コストが相対的に高くなるのは事実です。
*   **ドキュメントと運用ノウハウ**: AWSは国内シェアが非常に高く、日本語の公式ドキュメント、技術ブログ、QiitaやZennなどのコミュニティ記事が極めて豊富です。トラブルシューティングや運用ノウハウの検索性が高く、自己解決しやすい環境が整っています。

## 3. 技術的評価
Lambda + DynamoDBによるサーバーレスアーキテクチャと、従来の常駐サーバー（EC2やコンテナベースなど）との比較、および内部業務アプリにおける適性を評価します。

*   **常駐サーバー（EC2/コンテナ等）との比較**:
    *   **コストとスケーラビリティ**: サーバーレスはゼロスケールが可能であり、リクエストが少ないシステムではコストが極めて低く抑えられます。突発的なスパイクにも自動で対応できます。対して常駐サーバーはベースコストがかかりますが、一定以上の高いトラフィックが継続する環境ではコンテナ（ECS/EKS）やEC2の方がコストパフォーマンスに優れる場合があります。
    *   **パフォーマンス（コールドスタート）**: Lambdaはしばらく実行されないと、次の実行時にコンテナの起動オーバーヘッド（コールドスタート）が発生します。ミリ秒単位の安定したレイテンシが要求される場合は、常駐サーバーが有利です（プロビジョンド同時実行の設定で緩和は可能ですがコスト増になります）。
*   **内部業務アプリケーションへの適性**:
    *   内部業務アプリでは、**低メンテナンス性、低負荷の処理、入力エラーの防止**が重視されます。
    *   サーバー自体のOSパッチ当てやミドルウェアのバージョンアップなどの管理（パッチマネジメント）が不要になるため、Lambdaベースのサーバーレス構成は「低メンテナンス性」の観点では非常に優れています。
    *   一方で、データモデルの複雑なリレーションを扱う場合、NoSQLであるDynamoDBは設計が難しく（シングルテーブル設計など）、要件変更に弱いという特徴があります。複雑な検索や集計処理が多い業務システムの場合、RDS（PostgreSQLやMySQL等）とコンテナ環境の組み合わせ、あるいは要件によってはSQLiteやSQL Server ExpressなどのよりシンプルなRDBMSをオンプレミスや軽量なVM上に構成する方が、長期的な開発・保守の観点で適している（学習コストと運用負荷のバランスが取れる）ケースもあります。

## 4. AIコーディングエージェントとの親和性
AWS LambdaおよびDynamoDBの開発において、AIコーディングエージェントの親和性は比較的高いと言えますが、いくつか注意点があります。

*   **コードのモジュール化と単一責任**: Lambda関数は一般的に単一の機能（マイクロサービス）として小さく実装されるため、コンテキストサイズが限られるAIエージェントにとってコードの理解や生成、修正が容易です。
*   **IaCの生成**: AWS CDKやSAMなどのテンプレート生成は、公式ドキュメントや事例が豊富なため、AIは高品質なインフラ定義コードを出力できます。
*   **DynamoDBのクエリ構築**: DynamoDBの複雑なクエリやFilterExpression、UpdateExpressionの記述において、AIは強力なサポートを提供します。
*   **課題点**: 全体的なアーキテクチャの整合性（IAMの最小権限の設計、複数リソースにまたがるトラブルシューティングなど）をAI単独で完全に担保するのは難しく、人間による「AWSの最適な組み合わせを選択するセンス」やレビューが依然として重要になります。

## 5. 細かいパターン (Fine-grained patterns)
AWS Lambdaを活用した代表的な構成パターンを補足情報として記載します。

*   **API Gateway + Lambda**:
    *   最も一般的な同期APIの構成。REST APIまたはHTTP APIとして公開し、クライアントからのリクエストをLambdaで処理します。
*   **Lambda関数URL (Function URLs)**:
    *   API Gatewayを介さずに、直接Lambda関数に専用のHTTP(S)エンドポイントを付与するシンプルな構成。認証やルーティング要件が複雑でない軽量なWebhookや内部APIの作成に最適です。
*   **EventBridge / SQS + Lambda (非同期処理)**:
    *   メッセージキュー（SQS）やイベントバス（EventBridge）をトリガーとしてLambdaを起動する構成。システム間の疎結合化、バッチ処理、リトライ処理の実装において非常に堅牢なアーキテクチャとなります。
*   **S3トリガー + Lambda**:
    *   S3バケットへのファイルアップロードを検知してLambdaを起動する構成。画像のリサイズ処理や、アップロードされたCSVファイルのDynamoDBへの取り込み処理などに多用されます。
