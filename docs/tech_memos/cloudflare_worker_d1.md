# Cloudflare Worker + D1 技術メモ

## 1. ユーザーのメモ要約
Cloudflare WorkerとD1（SQLite）を組み合わせることで、JavaScript/TypeScriptを用いた標準的な構成のバックエンドを無料で構築・公開できる、非常に強力なサービスである。しかし、手放しで簡単に構築できるわけではなく、以下の課題が存在する。
- スクリプトのCPU時間に制限があるため、コードの書き方や実現できる処理に制約が生じる。
- AIコーディングエージェントにコードを生成させる際、AIが学習している少し古い情報をベースに出力しがちであり、Cloudflareの最新の設定やAPI仕様と合わずにハマる（つまずく）ことが多い。

## 2. AIの補足・明確化
Cloudflare Workersはサーバーレスの「エッジコンピューティング」環境であり、Node.jsなどの常駐プロセスとは異なる独自のランタイム（V8アイソレート）上で動作します。
また、Cloudflareのエコシステム（Wrangler CLI、D1のAPI、`wrangler.toml` の設定フォーマットなど）は進化が非常に早く、頻繁に破壊的変更や推奨されるベストプラクティスの更新が行われます。ユーザーが指摘した「AIが古い情報を出力する」という問題は、まさにこの進化のスピードと、AIモデルの事前学習データにおけるタイムラグに起因しています。なるべく最新の設定でスタートしたいという要件に対して、AIは学習済みの過去の設定を無意識に提示してしまうため注意が必要です。

## 3. 技術的評価
**常駐サーバー（Node.js + VPSなど）との比較**
一般的な常駐型サーバーであれば、長時間を要する重い処理（例: 大量のデータインポートや複雑なループ集計）もバックグラウンドでそのまま処理させることが可能です。一方、Cloudflare Workersの無料プランには「CPU時間（通常10ミリ秒）」の厳しい制限があります。そのため、CPUバウンドな処理を行うと即座に制限に引っかかり、エラーとなります。
ただし、「社内業務アプリケーションではメンテナンス性の高さや低負荷を優先し、データベースはSQLiteを推奨する」という当環境のアーキテクチャ選定方針においては、サーバー管理が不要（低メンテナンス）であり、SQLiteベースのD1を使用できるこの構成は非常に理にかなっています。CPU時間の制約さえ意識して適切に設計できれば、高いコストパフォーマンスと安定性を発揮します。

## 4. AIコーディングエージェントとの親和性
Cloudflareは頻繁なアップデートが行われるため、AIコーディングエージェント単独で最初から「完全な最新コードや設定」を生成させるのは困難な場合があります（AIが旧バージョンの `Wrangler` 設定や、古いバインディング方法を提案しがちです）。
そのため、エージェントとの親和性を高め、手戻りを防ぐためには以下のアプローチが必要です。
- **最新のプロジェクト雛形はCLIで生成する**: プロジェクトの初期設定（`wrangler.toml` や `package.json`）は最新の公式コマンド（`npm create cloudflare@latest` など）を用いて人間側、あるいはスクリプト側で生成し、AIには「この環境・制約の中でロジックを書く」ように指示する。
- **Web検索ツール（ブラウジング機能）の活用**: AIエージェントにコード実装を依頼する際、事前に公式ドキュメント（`developers.cloudflare.com`）を検索して最新のD1 API仕様を確認させるステップを組み込む。

## 補足情報: 細かいパターン（制約回避と最新実装のコツ）

### 1. CPU時間制限の回避パターン
Cloudflare WorkersのCPU時間は「外部通信やデータベース接続を待機している時間」はカウントされず、「実際にJSエンジンが処理を行っている時間」のみカウントされます。
- **D1側での処理**: データを全件取得してからJS側で集計・加工するのではなく、可能な限りSQL（D1）側で集計やフィルタリングを完了させ、JSの計算負荷を減らす。
- **重い処理の分割**: 大量のデータを処理する場合は、1回のWorker呼び出しで全て行わず、クライアント側でページネーションを用いてリクエストを小分けにするか、Cloudflare Queuesなどを併用して非同期タスクとして分割する。

### 2. D1データベースのバッチ処理パターン
D1に対して複数回のクエリを発行する際、都度リクエストを行うとオーバーヘッドが大きくなります。D1クライアントが提供するバッチ実行APIを使用することで、複数クエリを1回のリクエストで送信でき、効率よく処理できます。
```typescript
// 推奨されるバッチ処理の例
const statements = [
  env.DB.prepare('INSERT INTO users (id, name) VALUES (?, ?)').bind(1, 'Alice'),
  env.DB.prepare('INSERT INTO users (id, name) VALUES (?, ?)').bind(2, 'Bob'),
];
// 複数のクエリを1回のリクエストでバッチ送信する
const results = await env.DB.batch(statements);
```

### 3. wrangler.toml の最新設定の指定パターン
AIが古い `wrangler.toml` の書き方（例: v1時代の古いバインディング）を出力しないよう、設定ファイルには互換性フラグ（`compatibility_date`）を明記し、AIにも最新の仕様を前提とするよう伝えるのが効果的です。
```toml
# wrangler.toml の設定例
name = "my-d1-app"
main = "src/index.ts"
compatibility_date = "2024-02-28" # ここはなるべく最新の日付を指定する
compatibility_flags = ["nodejs_compat"] # 必要な場合

[[d1_databases]]
binding = "DB"
database_name = "my-database"
database_id = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
```
