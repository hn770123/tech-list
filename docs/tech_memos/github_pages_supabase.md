# GitHub Pages + Supabase 構成の技術メモ

## 1. ユーザーメモの要約

外部向けにサービスを展開する際、バックエンド機能を丸ごと肩代わりできる「Supabase」と、ファイルを置くだけでホスティング可能な「GitHub Pages」の組み合わせについての評価です。

*   **初期開発の最適解**: 突貫でサービスを構築するにあたり、非常に優れた組み合わせである。
*   **オンプレミス運用の可能性**: Supabaseはオープンソースであるため、プロジェクト全体をオンプレミスで運用（セルフホスト）することは一応可能。
*   **長期的な課題と移行**: オンプレミスでSupabaseを稼働させるための運用コストや手間を考慮すると、長期的にはよりシンプルな独自の構成に作り変えるべきである。
*   **適用ケース**: システムの使用頻度が高くなく、高いパフォーマンスも要求されないが、外部（インターネット）からのアクセスが必要な場合には、有力な候補となる。

## 2. AIの補足・明確化

上記のメモに対する補足事項です。

*   **SupabaseのBaaSとしての利点**: 認証（Authentication）、データベース（PostgreSQL）、ストレージ、リアルタイムリスナーなどの機能が最初から統合されているため、インフラのセットアップや初期のバックエンド実装工数を劇的に削減できます。これが「突貫サービスの最適解」たる所以です。
*   **GitHub Pagesの制約とメリット**: 静的ファイルのホスティング（HTML/CSS/JS）に特化しており、サーバー側の処理（Node.jsやPHPなど）を持たないため、セキュリティリスクが低く、デプロイメントが極めて容易です。Supabaseの提供するクライアントSDKと組み合わせることで、静的サイトながら動的なバックエンド連携を持つSPA（Single Page Application）を即座に公開できます。
*   **「長期的にはシンプルな構成に作り変える」の意図**: Supabaseは多機能で強力な反面、そのアーキテクチャは複数コンポーネント（PostgreSQL, GoTrue, Realtime, Storage, PostgREST等）の集合体です。これをDocker等で自前ホスト（オンプレ運用）する場合、保守・運用（アップデート追従、バックアップ、障害対応）の難易度が高くなります。そのため、要件が固まり次第、自社内で保守しやすいシンプルな常駐サーバー実装（例：Node.js + 単一のDBなど）にリプレイスする戦略は理にかなっています。

## 3. 技術的評価

本構成における技術スタックの評価を、常駐サーバー実装（バックエンド運用）の観点を中心に行います。

### 3.1. 構成の立ち上げ速度（短期評価）
*   **評価**: 非常に高い
*   **理由**: クラウド版SupabaseとGitHub Pagesを利用する前提であれば、インフラ構築の時間はほぼゼロです。フロントエンドのコードを書くことに集中できるため、PoC（概念実証）やプロトタイプ開発においてこれ以上の速度を出す構成は限られます。

### 3.2. オンプレミスでのセルフホスト運用（長期評価）
*   **評価**: 要注意（コスト高）
*   **理由**: Supabaseのセルフホストは公式にサポートされているものの、マイクロサービス・アーキテクチャに近い構成となるため、要求されるサーバースペックが高く、Dockerコンテナ群の管理に関する高度な知識が必要です。「使用頻度が高くなくパフォーマンスも不要」という要件に対して、システム構成がオーバースペックとなり、運用負荷（メンテナンスコスト）が跳ね上がるリスクがあります。

### 3.3. シンプルな自前サーバーへの移行容易性
*   **評価**: 設計次第で対応可能
*   **理由**: Supabaseは標準的なPostgreSQLをベースとし、APIはRESTful（PostgRESTベース）です。そのため、初期段階でフロントエンドとSupabase間の通信層（APIクライアントやデータアクセス層）を適切に抽象化しておけば、後日独自のシンプルな常駐サーバー（Node.js + SQLite/PostgreSQLなど）へバックエンドをすげ替える移行作業は比較的スムーズに行えます。

## 4. AIコーディングエージェントとの親和性

この構成でAIコーディングエージェントを利用した開発を行う場合の親和性です。

### 4.1. フロントエンド（GitHub Pages側）の実装
*   **親和性**: 非常に高い
*   **理由**: バックエンドの複雑なルーティングやサーバーサイドレンダリング（SSR）を考慮する必要がなく、純粋な静的サイト（HTML/JS）としての記述に集中できます。AIは静的なUIコンポーネントの生成や、BaaS（Supabase SDK）を呼び出すAPI連携のコード生成を非常に得意としています。

### 4.2. Supabase（バックエンド側）の設定・クエリ
*   **親和性**: 高い
*   **理由**: SupabaseのドキュメントやSDK（`@supabase/supabase-js`）は広く普及しており、AIの学習データに豊富に含まれています。RLS（Row Level Security）のSQLポリシーや、PostgreSQLのテーブル定義・関数作成のプロンプトに対しても、正確なコードを生成しやすい環境です。

### 4.3. 自前サーバーへの移行フェーズにおける親和性
*   **親和性**: 良好
*   **理由**: AIに対して「既存のSupabase API呼び出し部分を、新しい自作のNode.js APIサーバーに向けたFetchリクエストに置き換えて」といったリファクタリング指示を出すことで、移行作業の大部分を自動化・効率化することが可能です。

## 5. 補足情報（実装・運用の細かなパターン）

「突貫サービスから将来のシンプル構成への移行」を見据えた際のパターンの補足です。

### 5.1. バックエンド移行を見据えたアーキテクチャパターン
*   **Repositoryパターンの導入**: フロントエンドのコード内でSupabaseのSDK（`supabase.from('...')` など）を直接コンポーネントに散財させず、データアクセスクラス（Repository）として分離します。これにより、将来バックエンドを自前のAPIサーバー（例：社内の軽量な常駐サーバー）に変更する際、このRepository層を書き換えるだけで対応が完了します。

### 5.2. 段階的なオンプレ移行パターン（ハイブリッド構成）
*   **DBのみ自前化**: Supabaseの機能すべて（認証、ストレージ等）をオンプレに持ち込むのではなく、認証等はクラウドのSupabase（または他の軽量なサービス）に残し、クリティカルな業務データを持つPostgreSQL（あるいはSQLite等にダウングレード）のみを社内のシンプルな常駐サーバーに移行するアプローチ。段階的な移行と運用負荷の軽減を両立できます。
*   **BaaSからPaaS/IaaSへの移行**: GitHub Pages（静的ファイル）+ 自前のバックエンドサーバーという構成に移行した後、もし社内からしかアクセスさせない運用に変更するのであれば、フロントエンドとバックエンドを統合した単一の社内用常駐アプリケーション（Node.js等）に再構築することも、構成のシンプル化に寄与します。
